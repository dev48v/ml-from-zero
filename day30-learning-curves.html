📉 Learning Curves — Will More Data Help, or Do You Need a Better Model?

Your model scores 0.64 on held-out data. Do you collect more rows, or reach for a different model? The learning curve tells you which — before you waste a week.

A learning curve plots two scores as a function of the training-set size: the training score (how well the model fits the very data it learned from) and the validation score (how well it does on a held-out set it never saw). Everything below is computed for real, live in your browser — no pre-baked images. We draw samples from a hidden target function sin(1.5x) plus noise, fit a polynomial by least squares at each training size, and score both sets (R², floored at 0). The shape of the two curves is a diagnosis: if they meet at a low value the model is too simple (high bias / underfit — more data is useless); if a wide gap persists the model memorises its sample (high variance / overfit — more data helps). Drag the complexity, noise and regularisation knobs and watch the diagnosis change.

Complexity = degree of the fitted polynomial. Low degree can only draw gentle shapes (bias); high degree can wiggle through every point (variance). Ridge λ penalises big coefficients — it trades a little bias for a lot less variance.
LEARNING CURVE — R² vs training-set size train   validation
The shaded band is the gap = train − validation = the model's variance. Where the validation line plateaus is the bias floor (irreducible noise + how much the model is missing).
—
THE FIT — the model (sky) over the sample, vs the hidden target (grey dashed)
Underfit = a stiff line that ignores the wiggles. Overfit = a curve that chases every noisy dot.
How to read "will more data help?" Look only at the validation line at the right-hand edge. If it is still rising and sits well below the training line, you are on the steep part of the curve — more data will keep pushing it up (classic high variance). If it has gone flat and the two lines already touch, you have hit a wall that more rows cannot break — the ceiling is set by model bias, not by data volume, so you must change the model (more features, higher complexity, less regularisation), not the dataset. This single picture routinely saves teams from labelling thousands of rows that could never have moved the metric.

🔨 Build it yourself

    
Write the learning-curve loop from scratch — fit at growing training sizes, score train and validation each time — then reach for sklearn.model_selection.learning_curve, which does exactly this over cross-validation folds, and its sibling validation_curve for sweeping a single hyperparameter. The whole point is to turn "my model scores 0.64" into an action: more data, or a better model?

    
    
      
  1. 
        
    1
    The loop from scratch
    
        
    A learning curve is just a loop over training-set sizes. Hold out a fixed validation set, then for each size m fit on m rows and score both the training slice and the held-out set. Average several random subsets per size so the curve is smooth.
    
        
    const sizes = [4,6,9,12,16,20,26,32,40,50];
function learningCurve(deg, lam, noise){
  const pool = makePool(50, noise);     // training pool
  const val  = makePool(40, noise);     // fixed held-out set
  return sizes.map(m => {
    let tr=0, va=0, K=10;
    for (let k=0;k<K;k++){
      const s  = sample(pool, m);        // random m rows
      const w  = fitPoly(s.x, s.y, deg, lam);
      tr += clamp01(r2(s.y,  s.x.map(x=>predict(w,x))));   // TRAIN score
      va += clamp01(r2(val.y, val.x.map(x=>predict(w,x)))); // VALIDATION score
    }
    return { m, train: tr/K, val: va/K };
  });
}

    Fit & score — the pieces
    
        
    The fit is ridge polynomial least-squares — solve (ΦᵀΦ + λI)w = Φᵀy. The score is R² = 1 − SSres/SStot, floored at 0 so a wild overfit reads as a readable "0" instead of a huge negative. Note: the training score is scored on the same rows the model just fit, which is why it starts near 1 and can only fall.
    
        
    function r2(y, yhat){
  const m = y.reduce((s,v)=>s+v,0)/y.length;
  let ssr=0, sst=0;
  for (let i=0;i<y.length;i++){ ssr+=(y[i]-yhat[i])**2; sst+=(y[i]-m)**2; }
  return sst<1e-9 ? 0 : 1 - ssr/sst;         // fraction of variance explained
}
const clamp01 = v => v<0 ? 0 : (v>1 ? 1 : v);

// train score is optimistic (scored on data it fit) -> train >= val, always.
// the two curves are the whole diagnosis.

    scikit-learn: learning_curve
    
        
    In practice you never write the loop — learning_curve does it over cross-validation folds and returns train/test scores at each size. Pass any estimator and any scorer.
    
        
    import numpy as np
from sklearn.model_selection import learning_curve
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import PolynomialFeatures
from sklearn.linear_model import Ridge

model = make_pipeline(PolynomialFeatures(6), Ridge(alpha=0.0))

sizes, train_scores, val_scores = learning_curve(
    model, X, y,
    train_sizes=np.linspace(0.1, 1.0, 10),   # 10 growing subsets
    cv=5,                                     # 5-fold CV at each size
    scoring="r2")                             # any metric works

train_mean = train_scores.mean(axis=1)        # average over folds
val_mean   = val_scores.mean(axis=1)

    Plot & read the diagnosis
    
        
    Plot both means against size. Then read it mechanically: converged + low = bias; wide persistent gap = variance; converged + high = done.
    
        
    import matplotlib.pyplot as plt
plt.plot(sizes, train_mean, "o-", label="train")
plt.plot(sizes, val_mean,   "o-", label="validation")
plt.fill_between(sizes, val_mean, train_mean, alpha=.15)  # the gap = variance
plt.xlabel("training-set size"); plt.ylabel("R²"); plt.legend()

gap  = train_mean[-1] - val_mean[-1]
if   gap > 0.15:            print("HIGH VARIANCE -> more data / regularise / simpler")
elif val_mean[-1] < 0.6:   print("HIGH BIAS -> add features / complexity (NOT data)")
else:                      print("GOOD FIT")

    validation_curve — the other axis
    
        
    A learning curve varies the data size; a validation curve fixes the data and varies one hyperparameter. Same two-curve reading, but now the x-axis is complexity — you literally watch bias turn into variance as the knob turns, and pick the sweet spot where the validation score peaks.
    
        
    from sklearn.model_selection import validation_curve

degrees = range(1, 16)
train_s, val_s = validation_curve(
    make_pipeline(PolynomialFeatures(), Ridge(alpha=0)),
    X, y,
    param_name="polynomialfeatures__degree",
    param_range=degrees, cv=5, scoring="r2")

# left side (low degree): both low  -> bias
# right side (high degree): train high, val drops -> variance
best_degree = degrees[val_s.mean(axis=1).argmax()]   # peak of the val curve

    Act on the diagnosis
    
        
    The curve is only useful if it changes what you do next. Match the signature to the fix — and notice the two fixes point in opposite directions, which is exactly why guessing is so costly.
    
        
    # HIGH BIAS  (curves meet LOW, gap already closed)
#   -> add features, raise model complexity, reduce regularisation
#   -> DO NOT collect more data: the curves have already converged.
#
# HIGH VARIANCE (wide gap, validation still climbing)
#   -> collect more data   (the val curve is still rising)
#   -> or add regularisation / simplify the model / fewer features
#
# GOOD FIT (curves meet HIGH)
#   -> ship it; spend effort elsewhere.
