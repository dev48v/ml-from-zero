🎯 The Kernel Trick — Why You Never Build φ(x): Algorithms Touch Data Only Through Dot Products, and K(x,y)=φ(x)·φ(y) Computes That Dot Product in a Huge (Even Infinite) Feature Space Without Ever Forming φ

Day 12's SVM already quietly used an RBF kernel. This page is the standalone idea behind it. Some data is hopelessly non-linear in its own coordinates — concentric rings, XOR — and no straight line will ever split it. The fix is to map each point into a higher-dimensional feature space φ(x) where the classes do separate with a flat boundary. But φ can be enormous, even infinite-dimensional, so building it explicitly is impossible. The trick: linear algorithms only ever touch the data through dot products xᵢ·xⱼ. Replace every dot product with a kernel K(x,y)=φ(x)·φ(y) that you can evaluate directly — and you get the exact same answer as if you had lifted to φ, for the price of one cheap function call. Everything below is a real kernel perceptron: the Gram matrix and the decision boundary are computed live in JS, no library.

Pick a demo. Lift & separate trains a genuine kernel perceptron on non-linear data. With the linear kernel the boundary is a straight line and it fails on rings; flip to polynomial (1+x·y)² or the RBF/Gaussian kernel and watch the boundary curve into a circle that separates the classes — while the side panel shows the same rings becoming linearly separable once lifted to the feature coordinates (x₁², x₂²). Move your cursor over the plot to see K(cursor, x₀) and the classifier score f(cursor)=Σαᵢ K(xᵢ, cursor) update live. Kernel matrix (Mercer) builds the real n×n Gram matrix, computes its eigenvalues with a from-scratch Jacobi solver, and shows that valid kernels give a positive-semidefinite matrix — while a tempting-but-invalid "kernel" fails the test.

🔨 Build it yourself

    
A kernel is just a function of two points. The magic is that a whole family of linear algorithms can be rewritten to use only that function — never the coordinates. Here is a runnable from-scratch core: three kernels, a numeric proof that the polynomial kernel equals a dot product in a 6-D feature space we never build, the Gram matrix and its Mercer (PSD) check, and a full kernel perceptron whose decision boundary is genuinely computed. Everything the LOOK tab draws lives in these snippets.

    The kernels — each is one cheap function of two points
    
        
    A kernel K(x,y) is a similarity that secretly equals a dot product in some feature space. The linear kernel is the ordinary dot product; the polynomial adds all interaction terms up to degree d; the RBF (Gaussian) measures closeness and lives in an infinite-dimensional space. None of them ever forms φ.
    
        
    import numpy as np

def k_linear(x, y):                      # phi = (1, x1, x2, ...)
    return 1.0 + x @ y

def k_poly(x, y, d=2):                    # phi = all monomials up to degree d
    return (1.0 + x @ y) ** d

def k_rbf(x, y, gamma=4.0):              # phi lives in an INFINITE-dim space
    diff = x - y
    return np.exp(-gamma * (diff @ diff))

    Prove the trick — φ(x)·φ(y) equals (1+x·y)² without believing it
    
        
    For 2-D inputs the degree-2 polynomial kernel expands to a 6-D feature map φ(x)=(1, √2·x₁, √2·x₂, x₁², √2·x₁x₂, x₂²). Build φ explicitly, take the honest dot product, and confirm it matches the one-line kernel to machine precision. In real life you skip the left-hand side entirely.
    
        
    def phi_poly2(x):                        # the EXPLICIT degree-2 feature map (2D -> 6D)
    x1, x2 = x
    r2 = np.sqrt(2.0)
    return np.array([1.0, r2*x1, r2*x2, x1*x1, r2*x1*x2, x2*x2])

x = np.array([0.7, -0.3]);  y = np.array([-0.2, 0.9])
explicit = phi_poly2(x) @ phi_poly2(y)   # dot product in 6-D feature space
kernel   = (1.0 + x @ y) ** 2            # the trick: never build phi
print(explicit, kernel)                  # 1.0921...  1.0921...  -> identical
assert abs(explicit - kernel) < 1e-12    # the kernel trick, verified

    The Gram matrix & Mercer's condition
    
        
    Collect the kernel over every pair of points into the n×n Gram matrix. K is a valid kernel iff this matrix is symmetric positive semi-definite for any points — all eigenvalues ≥ 0. That is exactly the condition for it to be an inner product in some space. A "distance kernel" fails it.
    
        
    def gram(X, kernel):
    n = len(X)
    return np.array([[kernel(X[i], X[j]) for j in range(n)] for i in range(n)])

def is_valid_kernel(X, kernel, tol=1e-8):
    G = gram(X, kernel)
    assert np.allclose(G, G.T)            # must be symmetric
    eig = np.linalg.eigvalsh(G)           # Mercer: all eigenvalues >= 0
    return eig.min() >= -tol, eig.min()

print(is_valid_kernel(X, k_rbf))                 # (True,  ~0)      valid
print(is_valid_kernel(X, lambda a,b: np.linalg.norm(a-b)))  # (False, <0)  NOT a kernel

    The kernel perceptron — the boundary the LOOK tab draws
    
        
    The classic perceptron in dual form. Keep one coefficient αᵢ per training point; predict with f(x)=Σᵢ αᵢ K(xᵢ,x); on a mistake, nudge α for that point by its label. Swap the kernel and the very same loop learns a curved boundary — the whole point of kernelization.
    
        
    def train_kernel_perceptron(X, y, kernel, epochs=60):
    n = len(X)
    G = gram(X, kernel)                   # precompute K(xi, xj)
    alpha = np.zeros(n)
    for _ in range(epochs):
        errors = 0
        for t in range(n):
            score = np.dot(alpha, G[:, t])   # f(x_t) = sum_i alpha_i K(x_i, x_t)
            if np.sign(score) != y[t]:
                alpha[t] += y[t]             # dual update (no weight vector!)
                errors += 1
        if errors == 0:
            break                           # separated in feature space
    return alpha

def predict(x, X, alpha, kernel):
    return np.sign(sum(a * kernel(xi, x) for a, xi in zip(alpha, X) if a != 0))

    Draw the boundary & score it — linear fails, kernels win
    
        
    Evaluate predict on a grid of pixels to paint the decision regions, and measure accuracy on the rings data. The linear kernel can't beat ~50–60%; the polynomial and RBF kernels hit 100% because the rings are separable once lifted.
    
        
    Xr, yr = concentric_rings()               # class +1 inner, class -1 outer ring

for kernel in [k_linear, lambda a,b: k_poly(a,b,2), lambda a,b: k_rbf(a,b,4.0)]:
    alpha = train_kernel_perceptron(Xr, yr, kernel)
    acc = np.mean([predict(x, Xr, alpha, kernel) == t for x, t in zip(Xr, yr)])
    n_sv = int(np.sum(alpha != 0))
    print(kernel, f"acc={acc:.2f}  support_vectors={n_sv}")
# k_linear -> acc~0.55  (a line can't split concentric rings)
# k_poly2  -> acc 1.00  (a circle can)
# k_rbf    -> acc 1.00  (bumps around each cluster)

🎉 Day 42 of MachineLearningFromZero done.
The kernel trick is one of the most leveraged ideas in classical ML. Non-linear data becomes separable in a high-dimensional feature space φ(x), but φ is expensive or infinite, so you never build it: linear algorithms touch data only through dot products, and a kernel K(x,y)=φ(x)·φ(y) supplies that dot product directly. The polynomial kernel (1+x·y)ᵈ is a finite feature map you can write down; the RBF kernel is an infinite one you can't — yet both cost a single function call. A function is a legal kernel exactly when its Gram matrix is positive semi-definite (Mercer). Kernelize the perceptron and a straight rule in feature space becomes a curved boundary in the input plane. 👉 The same trick powers the SVM (Day 12), kernel PCA, and Gaussian Processes (Day 37).
